Tài liệu Kỹ thuật (Technical Blueprint)

Kiến trúc Hệ thống Khảo thí & Chấm thi Thông minh

Thiết kế kiến trúc hệ thống chuẩn mã nguồn, đảm bảo vận hành ổn định từ phiên bản chạy thử nghiệm (100 thí sinh) đến quy mô chính thức (1000 thí sinh đồng thời) mà không cần viết lại cốt lõi phần mềm.

👤 Đội ngũ Kỹ thuật 📅 Ngày 11 Tháng 6, 2026 🏷️ Phiên bản 1.1.0 ⚡ Trạng thái: Sẵn sàng triển khai

1. Tổng quan Kiến trúc Hệ thống

Hệ thống áp dụng mô hình kiến trúc phân tách hoàn toàn (Decoupled Architecture) giữa Frontend (Giao diện người dùng) và Backend (Dịch vụ API), liên kết với hệ thống hàng đợi bất đồng bộ xử lý chấm điểm AI.

🎯 Quy mô Demo (100 CCU)

Hạ tầng chuẩn hóa. Hệ thống chạy trên Docker với cơ sở dữ liệu PostgreSQL cục bộ, đảm bảo tính đồng nhất 100% về cấu hình, mô hình bảng và kiểu dữ liệu với môi trường Production chính thức.

⚡ Quy mô Chính thức (1000 CCU)

Nâng cấp phi tập trung. Tách biệt CSDL (PostgreSQL Cloud SQL), lưu trữ file (Google Cloud Storage), phân phối tĩnh qua CDN, và mở rộng số lượng worker chấm ngầm để chống nghẽn nghẹt hệ thống.

💡 Triết lý Kiến trúc: Mở rộng Hướng ngang (Horizontal Scalability)

Toàn bộ API được thiết kế dạng Stateless. Nghĩa là máy chủ API không lưu trạng thái đăng nhập hay dữ liệu phiên của người dùng. Khi quy mô sinh viên tăng lên 1000+, ta chỉ cần bật thêm các bản sao máy chủ API đứng sau bộ cân bằng tải (Load Balancer) mà không cần sửa bất kỳ dòng code nào.

2. Công nghệ Cốt lõi & Đánh giá Tối ưu

Bảng đánh giá lựa chọn công nghệ cho thấy sự phù hợp tuyệt đối cho cả hai giai đoạn: Demo và Nâng cấp chính thức.

Thành phần Công nghệ Lựa chọn Tối ưu cho Demo (100 CCU) Tối ưu cho Nâng cấp (1000 CCU)

Vận hành tài khoản Enterprise. Hàng đợi Celery tự động kiểm soát tần suất cuộc gọi (Rate-limit) tránh lỗi giới hạn RPM/TPM.

3. Sơ đồ Luồng Hoạt động

Sơ đồ dưới đây minh họa luồng đi của dữ liệu từ thiết bị của sinh viên qua các tầng xử lý và kết thúc tại dịch vụ chấm điểm AI.

3b. Luồng Xử lý Kỹ thuật các Tính năng Cốt lõi

Mô tả chi tiết các bước vận hành kỹ thuật (luồng đi dữ liệu) cho 4 chức năng cốt lõi của hệ thống MVP.

🔑 Luồng Xác thực & Phân quyền (JWT)

Đảm bảo tính bảo mật và phân quyền rõ ràng giữa Sinh viên, Giáo viên và Admin:

  1. Bước 1: Trình duyệt gửi request POST /api/v1/auth/login chứa mã sinh viên/mã giáo viên và mật khẩu.
  2. Bước 2: FastAPI truy vấn database, đối khớp mật khẩu băm (bcrypt hash).
  3. Bước 3: FastAPI tạo mã JWT Token chứa thông tin mã hóa (User ID, Role, thời gian hết hạn).
  4. Bước 4: Client lưu JWT vào localStorage hoặc Cookie an toàn của trình duyệt.
  5. Bước 5: Mọi request gọi API tiếp theo (lấy đề, nộp bài) đều phải đính kèm header Authorization: Bearer <JWT_TOKEN> để xác thực quyền.
🛡️ Luồng Giám sát & Báo cáo Vi phạm

Hạn chế gian lận trong thời gian thi trực tuyến nhờ các Web APIs tích hợp:

  1. Bước 1: Trình duyệt sinh viên lắng nghe sự kiện visibilitychange (chuyển tab) hoặc fullscreenchange (thoát toàn màn hình).
  2. Bước 2: Khi phát hiện hành vi thoát/chuyển tab, hook useProctoring.ts kích hoạt hiển thị cảnh báo đỏ và ghi nhận lỗi.
  3. Bước 3: Frontend gửi ngầm request POST /api/v1/exams/proctoring/log với loại vi phạm và mốc thời gian lên backend.
  4. Bước 4: FastAPI ghi dữ liệu vi phạm vào bảng proctoring_logs trong database để lưu vết vĩnh viễn.
  5. Bước 5: Màn hình giám thị (Teacher Dashboard) nhận thông báo vi phạm thời gian thực để đưa ra quyết định đình chỉ thi nếu cần.
💾 Luồng Tự động Lưu bài (Auto-save)

Phòng tránh mất mát bài làm của thí sinh khi gặp sự cố mất điện hoặc rớt mạng:

  1. Bước 1: Trong lúc thi, cứ mỗi 30 giây, một bộ timer chạy ngầm tại Frontend sẽ gom trạng thái bài làm hiện tại.
  2. Bước 2: Frontend đóng gói câu trả lời dưới định dạng JSON và gọi API POST /api/v1/submissions/{id}/autosave.
  3. Bước 3: FastAPI tiếp nhận JSON bài làm, thực hiện cập nhật đè (upsert) trường answers trong bảng submissions.
  4. Bước 4: Database phản hồi thành công, backend trả về kết quả 200 OK để đồng bộ trạng thái lưu trữ trên giao diện.
  5. Bước 5: Nếu thí sinh bị sập nguồn, khi mở lại máy và truy cập phòng thi, hệ thống sẽ tự động tải lại JSON bài làm cũ để thi tiếp.
🤖 Luồng Chấm thi Tự luận AI Asynchronous

Xử lý chấm bài tự luận nói/viết ngầm, tối ưu hóa hiệu năng hệ thống:

  1. Bước 1: Thí sinh ấn nộp bài, FastAPI cập nhật trạng thái bảng submission thành submitted và trả về kết quả cho client trong 0.1s.
  2. Bước 2: FastAPI đẩy một task chấm bài thi tự luận Nói/Viết vào hàng đợi Redis Broker.
  3. Bước 3: Celery Worker chạy ngầm bốc task từ Redis, lấy audio bài nói hoặc text bài viết từ database.
  4. Bước 4: Worker gửi yêu cầu kèm theo Prompt chấm chuẩn VSTEP/HSK và dữ liệu bài làm sang Gemini API.
  5. Bước 5: Nhận kết quả từ Gemini, Worker giải mã JSON (điểm, đánh giá lỗi phát âm/ngữ pháp) và lưu vào bảng grades, cập nhật trạng thái submission sang graded.

🎯 Công việc Frontend cần thực hiện đầu tiên (Bản Demo):

  • Thiết kế trang đăng nhập và lưu giữ JWT Token của sinh viên khi đăng nhập thành công.

  • Kết nối API và hiển thị audio cho các Part nghe trắc nghiệm.
  • Tích hợp hook nén âm thanh để đảm bảo ghi âm và gửi thử nghiệm file nói nhẹ dung lượng lên máy chủ.

6. Thiết kế Hệ thống Base Hỗ trợ Đa Ngôn Ngữ

Hệ thống được thiết kế ngay từ đầu để đáp ứng đồng thời cả Tiếng Anh (TOEIC, VSTEP B1) và Tiếng Trung (HSK) mà không cần cấu trúc lại lõi nghiệp vụ.

🔤 Cơ sở Dữ liệu UTF-8 Collation

Toàn bộ cơ sở dữ liệu PostgreSQL sử dụng bảng mã UTF8 và Collation đa ngôn ngữ chuẩn (lc_collate = 'en_US.UTF-8', lc_ctype = 'en_US.UTF-8'). Điều này đảm bảo lưu trữ chuẩn xác các ký tự đặc biệt của tiếng Việt có dấu, tiếng Anh, và các ký tự tượng hình của tiếng Trung (Simplified / Traditional) HSK mà không bị lỗi hiển thị font chữ (font artifacts).

⚙️ Dynamic Payloads via JSONB

Để đáp ứng sự khác biệt về nghiệp vụ (ví dụ: tiếng Trung cần lưu trữ Pinyin phiên âm, thứ tự nét viết nét vẽ; tiếng Anh cần link audio/script), các bảng questions và question_groups sử dụng cột metadata kiểu JSONB của PostgreSQL. Cấu trúc payload động này cho phép mở rộng thuộc tính tùy chỉnh theo ngôn ngữ mà không cần thay đổi schema vật lý của bảng.

Thành phần Thiết kế cho Tiếng Anh (TOEIC & B1) Mở rộng tương đương cho Tiếng Trung (HSK)

7. Chiến lược Chuyển đổi từ MVP lên Bản Chính thức

Đảm bảo mã nguồn chạy MVP (50-100 CCU) và bản chính thức (1000 CCU) là trùng khớp 100%. Sự khác biệt chỉ nằm ở cách cấu hình hạ tầng triển khai.

Dịch vụ Kiến trúc MVP (50 - 100 CCU) Kiến trúc Chính thức (500 - 1000 CCU)
Next.js Frontend Chạy Node container trên 1 VM VPS của trường. Build tĩnh (Static Export) deploy lên Google Cloud Storage + CDN toàn cầu (Vercel/Cloudflare). 0% tải cho server chính.
FastAPI API Chạy container trên cùng 1 VM VPS. Deploy lên Google Cloud Run (Serverless, tự động Scale out từ 1 lên 20+ instances).
PostgreSQL Database PostgreSQL chạy dưới dạng Docker container cục bộ, ánh xạ ổ cứng VPS để lưu trữ. Google Cloud SQL (PostgreSQL Managed) cấu hình High Availability, kết hợp PgBouncer làm Connection Pool chống nghẽn kết nối.
Queue & Cache Redis Container chạy cục bộ trên VPS. Google Cloud Memorystore (Redis Cluster Managed) chịu tải lớn, dự phòng thảm họa.
Media Storage Lưu trực tiếp vào thư mục VPS được ánh xạ (Local Docker Volumes). Google Cloud Storage (GCS) lưu file MP3 đề thi và Opus nộp bài của sinh viên, bảo mật qua Signed URLs.

8. Chi tiết Phân hệ Ra đề Tự động (Exam Generator)

Phân tích sâu cơ chế nạp liệu Word/Excel và thuật toán trộn đề sinh 100 bộ đề mẫu độc lập cho Tiếng Anh (TOEIC & VSTEP B1) trên base đa ngôn ngữ.

📥 Bộ Phân tách Dữ liệu (Parser Engine)

Backend sử dụng python-docx để bóc tách tệp câu hỏi gốc (.docx) và openpyxl để đọc bảng đáp án (.xlsx):

  • Phân tích Word (.docx): Nhận diện câu hỏi đơn và nhóm câu hỏi dựa trên cấu trúc Paragraph, Text Run và Tables. Nhận diện các thẻ đánh dấu đề để tách đoạn văn đọc hiểu chung (passage) hoặc script hội thoại.
  • Phân tích Excel (.xlsx): Tự động đọc danh sách câu hỏi, cột đáp án đúng (A/B/C/D), phân loại độ khó (Easy, Medium, Hard) và phần thi tương ứng.
  • Liên kết tài liệu: Ánh xạ tự động các file nghe trong thư mục audio thô với mã câu hỏi dựa trên quy tắc đặt tên file (ví dụ: LT2601_part1.mp3 tương ứng với exam_id = 2601, part = 1).

🎲 Thuật toán Trộn đề Ràng buộc (Constraint Generator)

Để tạo ra 100 bộ đề thi chuẩn hóa độc lập, thuật toán phải tuân thủ nghiêm ngặt ma trận ràng buộc:

  • Ràng buộc Độ khó (Difficulty Matrix): Định mức tỷ lệ 25% câu Dễ (Easy), 50% câu Trung bình (Medium), 25% câu Khó (Hard) cho từng phần thi (Part), cho phép sai số tối đa ±2 câu.
  • Ràng buộc Nhóm (Group Isolation): Đối với các câu hỏi đi theo đoạn văn (TOEIC Part 6-7, B1 Reading) hoặc đoạn hội thoại (TOEIC Part 3-4, B1 Listening), thuật toán bắt buộc phải trộn ở cấp độ Nhóm (Group-level), không được tách lẻ hoặc tráo đổi câu hỏi con ra khỏi đoạn văn/file nghe chung của nó.
  • Cân bằng Đáp án (Answer Balance): Phân bổ tỷ lệ đáp án đúng (A, B, C, D) cân bằng trong khoảng từ 20% đến 28% mỗi loại trên toàn bộ bài thi để loại bỏ các mẫu đoán mò.
  • Tránh lặp chủ đề (Topic Diversity): Kiểm soát không cho xuất hiện cùng 1 chủ đề từ vựng (ví dụ: Email thương mại, Quảng cáo, Vé máy bay) quá 2 lần liên tiếp.

9. Quy chuẩn Thiết kế Specs & Tests (Harness Engineering)

Áp dụng triết lý Harness Engineering nhằm thiết lập môi trường giàn thử nghiệm (Test Harness) tự động, giúp kiểm định chất lượng đề thi được sinh ra trước khi đưa vào phòng thi chính thức.

📋 Quy chuẩn Kỹ thuật (Specifications)

Từng phân hệ nghiệp vụ bắt buộc phải tuân thủ các mã Spec định nghĩa sẵn trong hệ thống:

  • SPEC-PARSE-001 (Tính toàn vẹn dữ liệu): Bộ phân tách tệp (Parser Engine) sau khi chạy không được để lại bất kỳ bản ghi câu hỏi nào bị khuyết trường correct_answer hoặc khuyết danh sách phương án options.
  • SPEC-MATRIX-002 (Tỷ lệ ma trận độ khó): Mọi đề thi sinh ra phải đảm bảo số lượng câu Easy nằm trong khoảng [45, 55], Medium [95, 105], Hard [45, 55] (đối với TOEIC 200 câu).
  • SPEC-ISOLATE-003 (Ràng buộc nhóm): Khóa ngoại group_id của các câu hỏi con thuộc nhóm bắt buộc phải tồn tại trong bảng question_groups của đề thi đó. Không chấp nhận tình trạng câu hỏi bị mồ côi (orphaned question).
  • SPEC-COLLATE-004 (Định dạng UTF-8): Tất cả các văn bản câu hỏi, câu trả lời tự luận không được chứa các ký tự lỗi mã hóa (ví dụ: ).

🏗️ Giàn Thử nghiệm (Test Harness Architecture)

Môi trường thử nghiệm độc lập được thiết lập tự động để kiểm thử code sinh đề và chấm đề:

  • Test Scaffolding (Mock Files): Thư mục tests/fixtures/ chứa các tệp Word/Excel mẫu chuẩn và các tệp bị cố ý làm hỏng cấu trúc (thiếu cột đáp án, lỗi font tiếng Trung, thừa đáp án E, v.v...) để kiểm tra độ chịu lỗi của Parser.
  • Automated Execution (Pytest Harness): Bộ công cụ chạy test tự động. Khi chạy lệnh kiểm thử, Test Runner sẽ khởi tạo DB kiểm thử tạm thời (PostgreSQL test database hoặc Sqlite in-memory), nạp dữ liệu mockup, chạy thuật toán trộn và kiểm tra toàn bộ mã Spec.
  • Programmatic QA Gate (Exam Validator): Một validator script tự động chạy thử nghiệm giải đề giả lập (Simulation of 10,000 runs) để verify xem hệ thống chấm điểm có trả về kết quả khớp 100% với key hay không. Bộ đề thi chỉ được đổi trạng thái is_published = True sau khi đã vượt qua QA Gate này.

⚙️ Mẫu kịch bản kiểm thử tích hợp (Integration Test Harness Example):

# File: backend/tests/test_harness/test_exam_generator.py
import pytest
from app.services.toeic_generator import ExamGenerator
from app.schemas.exam import ExamCreateSchema

def test_generator_matrix_adherence(db_session):
    """
    Harness Test verifying SPEC-MATRIX-002:
    Ensures that the generated exam adheres strictly to the 25% - 50% - 25% difficulty matrix.
    """
    # 1. Setup Mock Question Bank
    generator = ExamGenerator(db=db_session)
    
    # 2. Execute Generator
    new_exam = generator.generate_single_exam(title="Harness Test Exam", language="english")
    
    # 3. Assertions (Programmatic QA Gate Checks)
    easy_count = sum(1 for q in new_exam.questions if q.difficulty == "easy")
    medium_count = sum(1 for q in new_exam.questions if q.difficulty == "medium")
    hard_count = sum(1 for q in new_exam.questions if q.difficulty == "hard")
    
    # Verify exact TOEIC 200 questions count
    assert len(new_exam.questions) == 200, "SPEC-MATRIX-002: Exam must have exactly 200 questions"
    
    # Verify difficulty tolerances
    assert 45 <= easy_count <= 55, f"SPEC-MATRIX-002: Easy questions count {easy_count} out of bounds"
    assert 95 <= medium_count <= 105, f"SPEC-MATRIX-002: Medium questions count {medium_count} out of bounds"
    assert 45 <= hard_count <= 55, f"SPEC-MATRIX-002: Hard questions count {hard_count} out of bounds"
    
    # Verify answer options distribution (A, B, C, D frequency should be between 20% and 30% each)
    correct_answers = [q.correct_answer for q in new_exam.questions if q.correct_answer]
    for option in ['A', 'B', 'C', 'D']:
        option_ratio = correct_answers.count(option) / len(correct_answers)
        assert 0.20 <= option_ratio <= 0.30, f"SPEC-MATRIX-002: Option {option} distribution is skewed: {option_ratio:.2f}"

Lệnh thực thi chạy test harness tự động tại terminal backend:

# Chạy pytest kèm báo cáo đo độ bao phủ code (code coverage)
pytest tests/test_harness/ --cov=app/services --cov-report=term-missing

10. Chiến lược Chịu tải nâng cấp lên 1000 sinh viên

Phân tích sự khác biệt về cấu hình vận hành để chứng minh hệ thống dễ dàng chịu tải khi tăng gấp 10 lần quy mô người dùng.

⚠️ Các rủi ro sập hệ thống khi tăng tải
  • Nghẽn băng thông: 1000 học sinh tải file ghi âm bài Nói (WAV thô ~20MB) cùng một lúc làm sập đường truyền của trường.
  • Sập Database: 1000 kết nối đồng thời từ API mở vào Database làm Database quá tải RAM/CPU.
  • Hết hạn mức AI: Gửi 1000 yêu cầu chấm thi tự luận dồn dập vào Gemini API cùng lúc bị lỗi trả về `429 Too Many Requests`.
🛠️ Cách thiết kế này giải quyết triệt để
  • N Nén cục bộ tại Client: `useMediaRecorder` nén sang Opus chỉ còn ~1.5MB trước khi gửi. Giảm 90% băng thông mạng.
  • Connection Pooling: Dùng `PgBouncer` giới hạn kết nối thực tế vào PostgreSQL chỉ tối đa 50 kết nối an toàn.
  • Hàng đợi điều phối AI: Hàng đợi Redis xếp 1000 bài thi Nói/Viết lại. Celery Worker chấm tuần tự 20 bài/phút giúp an toàn tuyệt đối, không lo lỗi API Gemini.
