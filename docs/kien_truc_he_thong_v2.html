1.Tóm tắt điều hành

Mục tiêu: xây dựng hệ thống ra đề tự động và chấm thi tự động bằng AI cho chứng chỉ ngoại ngữ: sinh 100 bộ đề TOEIC (2 kỹ năng Nghe–Đọc, 200 câu) và 100 bộ đề B1 VSTEP (4 kỹ năng, gồm tự luận Viết–Nói chấm bằng Gemini) từ ngân hàng đề thô của đối tác (~30 bộ Word/Excel/MP3 mỗi loại), phục vụ thi trực tuyến có giám sát.

Hai trục chiến lược — quyết định toàn bộ kiến trúc

Trọng tâm hiện tại: trong 2 trụ cột của tiếng Anh (Ra đề và Chấm đề AI), dự án đang tập trung vào phân hệ Ra đề — được đào sâu ở mục 5 (Parser nạp liệu → Ngân hàng câu hỏi → Thuật toán sinh đề). Chấm AI được phủ ở mức kiến trúc tại mục 6.

Nguyên tắc Harness Engineering

Mỗi bảo chứng kiến trúc = 1 mã SPEC + tiêu chí chấp nhận + test tự động. Tài liệu này không chỉ mô tả thiết kế mà cam kết thiết kế: 23 spec trong mục 9 được đối chiếu tự động với bộ test pytest (truy vết 2 chiều). Trạng thái hôm nay: 8 active (đang đạt), 7 gap (đã đặc tả, code chưa đạt — test xfail tự phát hiện khi sửa xong), 8 planned (subsystem chưa triển khai).

2.Đánh giá Tech Stack (trung thực)

Stack đã chọn đúng hướng cho bài toán: API bất đồng bộ + hàng đợi chấm AI + frontend tách rời là mô hình chuẩn cho hệ thống khảo thí. Bảng dưới là đánh giá trung thực từng lớp, đối chiếu với hiện trạng code thật trong repo (không phải mô tả lý tưởng):

LớpLựa chọnPhán quyếtGhi chú trung thực (hiện trạng repo)
FrontendNext.js + TypeScript + Tailwind v4 Giữ + bổ sung Repo thực tế đang ở Next.js 16.2.7 / React 19 (kế hoạch ghi 15 — cập nhật tài liệu, không cần hạ cấp). Trang phòng thi TOEIC + hooks proctoring/ghi âm đã chạy. Thiếu: lớp API client dùng chung (đang fetch thô, hardcode localhost:8000) và i18n (chuỗi UI đang hardcode tiếng Việt).
BackendFastAPI + Pydantic v2 Giữ Khung API, models, services sinh đề/chấm đề TOEIC đã có. Thiếu: JWT auth (SECRET_KEY đã khai trong config nhưng chưa có endpoint đăng nhập/middleware phân quyền).
DatabasePostgreSQL + SQLAlchemy 2 Giữ + bổ sung Đúng lựa chọn (UTF-8, JSON/JSONB, connection pool). Thiếu Alembic migration — hiện dùng create_all() tự tạo bảng, sẽ bế tắc ngay khi cần thêm cột (source_question_id, meta…) trên DB có dữ liệu. SQLite fallback chỉ dùng cho dev/test; lưu ý hành vi JSON và concurrency lệch so PostgreSQL.
Hàng đợiCelery + Redis Giữ + nối dây Đã cấu hình (app/core/celery.py) và đã có task grade_submission_task (app/workers/tasks.py) — nhưng API chưa gọi: endpoint submit đang chấm đồng bộ. Phải nối queue trước khi có bài Viết/Nói thật (bảo chứng SPEC-GRADE-003).
AIGemini 3.1 Pro (chấm) + 3.5 Flash (phụ trợ) Giữ + cảnh báo Tích hợp đã có kèm mock fallback an toàn khi thiếu key (SPEC-GRADE-004 ✓). Hai cảnh báo: (1) SDK google-generativeai đã ngừng hỗ trợ — lên kế hoạch chuyển sang google-genai; (2) rate limit 429 khi 500–1000 bài dồn cùng lúc → bắt buộc đi qua hàng đợi có điều tốc + retry, không gọi trực tiếp từ API.
Storage file(chưa chọn rõ) Bổ sung Audio MP3 đề thi + file ghi âm bài nói cần storage abstraction: local FS (MVP) ↔ Google Cloud Storage (Prod), chọn qua STORAGE_BACKEND trong .env. Hiện audio_url là chuỗi tự do, chưa có lớp lưu trữ.

Kết luận: không đổi công nghệ nào. Bốn việc bổ sung bắt buộc trước khi lên Production: ① Alembic migration, ② JWT auth, ③ nối Celery vào API submit, ④ storage abstraction. Cả bốn đều đã có spec tương ứng ở mục 9 để theo dõi.

3.Kiến trúc tổng thể

Mô hình Decoupled + Stateless: frontend và backend tách rời hoàn toàn; API không giữ trạng thái phiên thi trong RAM — mọi trạng thái nằm trong PostgreSQL, nên có thể nhân bản API/worker theo chiều ngang mà không sửa code.

Nguyên tắc bất biến

Nguyên tắcÝ nghĩa thực tế
API statelessKhông session trong RAM; JWT mang danh tính, DB giữ trạng thái bài thi → nhân bản N instance API không cần sticky session.
Trạng thái thi trong DBAuto-save 30s ghi thẳng vào DB → thí sinh rớt mạng/đổi máy vẫn khôi phục được; không mất bài khi 1 instance API chết.
Worker idempotentTask chấm chạy lại không nhân đôi điểm (upsert theo submission_id); retry an toàn khi Gemini lỗi 429/5xx.
Cấu hình từ envSPEC-SCALE-001: mọi URL hạ tầng từ .env — nền tảng của chiến lược MVP → Prod không sửa code.

4.Thiết kế CSDL & quyết định đa ngôn ngữ

5 quyết định đa ngôn ngữ (định hình base cho cả EN lẫn CN)

Quyết địnhLý do / cách làm
1. language + exam_type là DỮ LIỆU Không nhánh code theo ngôn ngữ. Một hàm sinh đề, một bộ bảng — đề TOEIC, VSTEP hay HSK chỉ khác giá trị cột.
2. Metadata đặc thù ngôn ngữ nằm trong JSON Pinyin, hsk_level (CN) hay vocabulary_level (EN) đặt trong options/meta JSON — không thêm cột riêng cho từng ngôn ngữ. Đã chứng minh round-trip CJK nguyên vẹn bằng test (SPEC-I18N-001 ✓).
3. UTF-8 toàn trình PostgreSQL encoding UTF8; mọi file đọc/ghi encoding="utf-8"; spec quét tự động ký tự lỗi U+FFFD (SPEC-COLLATE-004 ✓).
4. Blueprint mỗi chứng chỉ là 1 BẢN GHI dữ liệu Cấu trúc đề (part → số câu, ma trận độ khó, ràng buộc) lưu ở bảng blueprints — thêm HSK = INSERT 1 dòng, không sửa generator (chi tiết mục 5.3).
5. Prompt chấm AI là FILE dữ liệu theo ngôn ngữ prompts/en/vstep_writing.json, prompts/cn/hsk_writing.json… — đổi rubric không cần deploy code.
⚠ Nợ kỹ thuật đã ghi nhận (SPEC-GRADE-002 — gap): grades.score_speaking đang chứa điểm Listening và score_writing chứa điểm Reading (mượn trường vì TOEIC không có Nói/Viết). Khi B1 VSTEP vào (có Nói/Viết thật) sẽ xung đột ngữ nghĩa. Phương án: Alembic thêm score_listening/score_reading, migrate dữ liệu, trả score_speaking/writing về đúng nghĩa. Test xfail đã cài sẵn — tự báo khi sửa xong.

5.Deep dive: Phân hệ Ra đề TRỌNG TÂM HIỆN TẠI

Phân hệ Ra đề là một pipeline 3 giai đoạn: nạp dữ liệu thô vào ngân hàng (Parser) → quản trị/duyệt ngân hàng (Bank) → sinh đề theo ràng buộc (Generator). Sản phẩm cuối: 100 bộ đề TOEIC + 100 bộ đề B1 VSTEP chuẩn hóa, đã qua kiểm định tự động.

5.1 Giai đoạn 1 — Parser/Ingestion Engine chưa triển khai — đặc tả mục tiêu

Kiểm kê dữ liệu đầu vào thực tế

Bộ dữ liệuFile nguồnLưu ý kỹ thuật parser
TOEIC Listening LT2601.docx → LT2629.docx (chỉ số lẻ, ~15 bộ) + Excel đáp án + thư mục "File nghe tong hop" (MP3) Liên kết MP3 ↔ nhóm câu theo quy ước tên file; thiếu file nghe → import fail (SPEC-PARSE-002).
TOEIC Reading CDR TOEIC - RT2601.doc → RT2615.doc (~15 bộ) + Excel đáp án .doc là định dạng nhị phân — python-docx không đọc được, bắt buộc bước convert sang .docx (LibreOffice headless) trước khi parse.
B1 VSTEP ~30 bộ mã 2601–2630: LB1.xxxx.docx (Nghe) + EB1 - xxxx.docx (Đọc & Viết) + Speaking Cards + phiếu tiêu chí chấm Phiếu tiêu chí chấm được số hóa thành prompt chấm AI (prompt-as-data, mục 4); Speaking Cards thành câu hỏi type="speaking".
Vòng đời dữ liệu nạp vào: mọi item từ parser vào bank ở trạng thái draft → giáo viên/admin duyệt trên giao diện quản trị → approved mới đủ điều kiện vào đề (SPEC-BANK-001). Điều này chặn lỗi parser "lọt lưới" vào đề thi thật.

5.2 Giai đoạn 2 — Quản trị Ngân hàng câu hỏi

Ngân hàng = các bản ghi Question/QuestionGroup có exam_id IS NULL. Vòng đời 3 trạng thái:

Trạng tháiÝ nghĩaĐược vào đề?

API quản trị đề xuất (router /api/v1/bank, yêu cầu role admin/teacher):

Gap đã ghi nhận (SPEC-BANK-001 — xfail): generator hiện chưa lọc status='approved' khi chọn câu từ bank. Vô hại lúc này (bank test toàn approved) nhưng phải vá trước khi parser đổ dữ liệu draft vào. Ngược lại, SPEC-BANK-002 ✓ — sinh đề chỉ CLONE, đã chứng minh bank bất biến trước/sau khi sinh.

5.3 Giai đoạn 3 — Thuật toán sinh đề

Blueprint-as-data

Cấu trúc mỗi chứng chỉ là một bản ghi dữ liệu trong bảng blueprints, generator chỉ đọc và thực thi. Blueprint TOEIC (đúng số liệu kế hoạch nghiệp vụ):

{
  "exam_type": "TOEIC", "language": "EN", "total": 200, "duration_minutes": 120,
  "difficulty_matrix": { "easy": 50, "medium": 100, "hard": 50, "tolerance": 2 },
  "parts": [
    { "part": 1, "mode": "standalone", "count": 6,  "difficulty": {"easy": 2, "medium": 3,  "hard": 1} },
    { "part": 2, "mode": "standalone", "count": 25, "difficulty": {"easy": 6, "medium": 13, "hard": 6} },
    { "part": 3, "mode": "group", "groups": 13, "q_per_group": 3, "g_difficulty": {"easy": 3, "medium": 7, "hard": 3} },
    { "part": 4, "mode": "group", "groups": 10, "q_per_group": 3, "g_difficulty": {"easy": 2, "medium": 5, "hard": 3} },
    { "part": 5, "mode": "standalone", "count": 30, "difficulty": {"easy": 8, "medium": 15, "hard": 7} },
    { "part": 6, "mode": "group", "groups": 4, "q_per_group": 4, "g_difficulty": {"easy": 1, "medium": 2, "hard": 1} },
    { "part": 7, "mode": "group_variable", "target_questions": 54, "difficulty": {"easy": 9, "medium": 28, "hard": 17} }
  ],
  "constraints": { "answer_balance": [0.20, 0.28], "topic_max_share": 0.20, "cross_exam_overlap_max": 0.40 }
}

Blueprint B1 VSTEP cùng schema: Listening 35 câu (3 phần), Reading 40 câu (4 đoạn × 10), Writing 2 task (type="writing": email 120 từ + luận 250 từ), Speaking 3 phần (type="speaking"). Blueprint HSK sau này: thêm 1 bản ghi — generator không đổi một dòng nào.

Kiến trúc thuật toán mục tiêu

generate_exam(blueprint, seed) →
  1. KIỂM TỒN KHO   bank đủ approved cho từng (part × độ khó)?  thiếu → raise (GEN-006)
  2. CHỌN           random.Random(seed) chọn câu/nhóm theo blueprint,
                    ưu tiên ít dùng (cân bằng overlap GEN-004), ràng buộc topic (GEN-003)
  3. CÂN BẰNG ĐÁP ÁN hoán vị vị trí options + reference_answer để đạt 20–28% (GEN-002)
  4. CLONE           ghi bản sao vào đề, kèm source_question_id truy vết nguồn
  5. POST-CHECK      chạy VALIDATOR trên đề vừa sinh: toàn bộ ràng buộc ở bảng trên
                     → fail: rollback transaction, báo lỗi; đạt: commit + is_active
Validator dùng chung với test harness: bước 5 và các test SPEC-GEN-* chạy cùng một bộ hàm kiểm tra (module app/services/exam_validator.py — đề xuất). Một nguồn logic duy nhất: đề sinh ở production được kiểm bằng đúng thước đo mà CI dùng để kiểm generator.

6.Thi & Chấm AI (mức kiến trúc)

Phòng thi trực tuyến (đã có khung chạy)

  • Làm bài: đếm ngược theo duration_minutes, tự nộp khi hết giờ; palette điều hướng câu hỏi; đánh dấu xem lại; audio player cho phần Nghe.
  • Auto-save 30 giây: gom trạng thái bài làm POST về backend, upsert theo submission — rớt mạng/đổi máy không mất bài.
  • Giám sát (useProctoring): phát hiện chuyển tab + thoát fullscreen (đếm vi phạm), webcam 320×240 hiển thị nổi; log vi phạm kèm timestamp gửi về backend.
  • Ghi âm bài nói (useMediaRecorder): nén tại trình duyệt sang Opus/WebM (~1.5MB/bài thay vì ~20MB WAV — giảm 90% băng thông khi 1000 thí sinh nộp cùng lúc), fallback codec đa trình duyệt.
Hiện trạng frontend: trang phòng thi đang chạy trên mock data và chấm fallback cục bộ khi API tắt; useMediaRecorder đã viết xong nhưng chưa gắn vào màn thi (chờ phần Speaking B1). Cần lớp API client dùng chung + biến môi trường NEXT_PUBLIC_API_URL thay cho hardcode localhost.

Luồng chấm thi — trắc nghiệm tức thì, tự luận bất đồng bộ

Quy đổi điểm & an toàn AI

  • TOEIC: % câu đúng mỗi kỹ năng tra bảng toeic_scoring_table.json → 5–495/kỹ năng, tổng 10–990. Bảng đã được kiểm thuộc tính tự động: đủ 101 mốc, đơn điệu không giảm, đúng biên (SPEC-GRADE-001 ✓).
  • B1 VSTEP Viết/Nói: rubric trong prompt-as-data (prompts/en/vstep_*.json); kết quả JSON {score, feedback, grammar_errors / transcription, pronunciation_issues} lưu vào grades.feedback_*.
  • Degrade an toàn: thiếu key/Gemini lỗi → mock có cấu trúc, không ném exception, không gọi mạng (SPEC-GRADE-004 ✓) — phòng thi không bao giờ sập vì AI sập; bài chờ chấm lại qua retry.
